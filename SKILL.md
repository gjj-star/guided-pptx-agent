# 引导型 PPT 生成

不直接生成 PPT。先用结构化提问帮你理清需求，确认后输出一份可执行规格。将规格交给任何 PPT 生成工具即可执行。

## 触发条件

用户消息包含以下任一关键词：
- 做 PPT / 生成 PPT / 帮我做演示文稿 / 准备分享材料 / 汇报 PPT
- presentation / slides / PowerPoint / deck

**不触发的情况**：用户已提供完整规格参数（配色、页数、内容大纲、受众、场景）。此时直接进入规格输出阶段，跳过访谈。

## 提问格式

Q1-Q3、Q5-Q8 是**结构化选择题**，以分组卡片式输出（模拟 AskUserQuestion）：

```
Q1: 核心目的是什么？
(A) 汇报  (B) 说服(推荐)  (C) 教学  (D) 品牌
```

Q4 是**开放式问题**，单独以文字形式提问，不给选项框架。Q4 的后续追问也以文字进行，不套用卡片格式。

T1、T2 是**条件触发的结构化题**，触发时以卡片式输出。

## 原则

1. 每轮对话问 3-4 个问题，每个问题给 2-6 个选项并标注推荐项。用户只需回答选项标签或序号，不需要写任何 prompt。
2. 只问不能推断的问题。能从已确认信息推断的维度，不再提问。
3. 全部问题完成后再汇总确认。用户说"确认"或"可以"后才输出最终规格。
4. 如果用户中途说"全部用默认"，跳过剩余问题，按推荐方案生成规格。

### 非标准答案处理

当用户的回答不在预设选项 A/B/C/D 中时（如 Q1 回答"小组作业"、Q2 回答"老师同学"），按以下规则映射：

1. 提取用户回答中的关键词
2. 匹配到语义最接近的预设选项
3. **必须向用户确认映射结果**，格式：「你说的"小组作业"，我理解为——目的≈汇报(A)，受众≈一线团队(C)，场景≈投影(A)。对吗？还是需要调整？」
4. 用户确认后再继续。如果用户说"不对"，追问具体偏差并重新映射。

映射参考表：

| 用户常见说法的可能映射 |
|------|
| "小组作业""课程展示" → 目的=A或C, 受众=C或B, 场景=A |
| "给老师看""答辩" → 目的=A或B, 受众=B或D, 场景=A |
| "内部分享""周会" → 目的=A, 受众=A或B, 场景=A |
| "路演""融资" → 目的=B, 受众=D, 场景=A |
| "发群里""邮件发" → 场景=B |
| 无法判断时 → 优先选推荐默认值，并向用户说明理由。

## 交互流程

### 第 1 轮：锚点四问

先同时输出这四个问题，等待用户回答后再进入下一轮。

**Q1: 核心目的是什么？**
(A) 汇报进展或同步信息
(B) 说服决策者（批准预算/通过方案） **← 推荐**
(C) 培训或教学
(D) 品牌展示或对外宣传

**Q2: 受众是谁？**
(A) 高层决策者（C-suite/VP/董事会） **← 推荐**
(B) 中层管理者或项目负责人
(C) 一线团队或执行者
(D) 外部客户或合作伙伴

**Q3: 最终交付场景？**
(A) 会议室投影或大屏 **← 推荐**
(B) 邮件或微信发送，对方自行阅读
(C) 需要打印成纸质材料
(D) 手机或平板浏览

**Q4: 只能让观众记住的一件事是什么？**

用一句话说出来。这不是标题，不是主题，是**结论**——你要让观众带走的那一句话。

好的核心信息：「我们的方案能在 6 个月内将客服成本降低 40%。」
不好的（这是主题，不是结论）：「AI Agent 技术分享」

以纯文本提问，不给选项框架。

**Q4 多级引导（当用户回答不出或给出的是主题而非结论时，逐级推进）：**

Level 1 — 用户给出主题而非结论（如"关于AI Agent的分享""小组作业展示"）：

> 你说的"AI Agent"是主题，但核心信息需要是一个**结论**。你的 PPT 最终想让观众相信什么、同意什么、或者决定做什么？
> 比如：不是"关于AI Agent"，而是"AI Agent 已经可以在 XX 场景中替代人工，企业应该立刻开始试点"。

Level 2 — 用户说"结论比较复杂"或"不止一个结论"：

> 那先不纠结用词。你想让观众感受到的是什么？选一个方向：
> (A) 两组/两方案之间的差异
> (B) 某个现象的原因或逻辑
> (C) 一个具体的行动建议
> (D) 一个趋势判断或预测

Level 3 — 用户选了方向后，给出该方向的填空模板：

- 如果选(A)差异：「[A]和[B]在[维度]上有本质不同，具体表现为[数据/事实]」
- 如果选(B)原因：「[现象]的根本原因不是[X]，而是[Y]」
- 如果选(C)建议：「我们建议在[时间]内做[行动]，预期可以带来[效果]」
- 如果选(D)预测：「[趋势]将在[时间]内成为主流，[谁]需要提前准备」

帮用户填入具体内容，提炼成一句 30 字以内的完整结论。

Level 4 — 用户无法填入内容：

> 看来内容还没准备到能提炼结论的程度。没关系——我们先跳到素材环节（Q5），你告诉我有多少已成型的内容，我们根据素材倒推核心信息。结论可以最后再定。

此时将 Q4 标记为"待定"，先进入 Q5。全部问题完成后，在汇总确认前回到 Q4，用已确定的素材反推结论。

### 第 1 轮推断规则

根据第 1 轮答案，决定后续推荐方向（不要在提问中展示推断过程）：

- 目的=B → 叙事推荐"结论先行"
- 目的=C → 叙事推荐"渐进呈现"
- 目的=D → 叙事推荐"故事弧线"
- 受众=A → 信息密度=极简，字号>=20pt
- 场景=A → 默认格式=PPTX，16:9，字号>=18pt，高对比度
- 场景=C → 避免大面积深色背景

### 第 2 轮：素材与约束

**Q5: 内容素材现在是什么状态？**
(A) 已有完整文稿或逐页大纲 **← 推荐**
(B) 只有关键数据和要点，需要协助组织
(C) 只有初步想法，需要从零梳理
(D) 已有旧版 PPT，需要翻新或参考

**Q6: 品牌视觉方面有什么约束？**
(A) 无约束，可以自由设计 **← 推荐**
(B) 有公司模板（请提供文件路径）
(C) 有严格品牌手册，必须遵守（请描述主色调和字体）
(D) 有参考 PPT 觉得方向对的（请提供文件路径）

**Q7: 页数和时长有限制吗？**
(A) 无硬性限制，你根据内容推荐 **← 推荐**
(B) 10 分钟以内，建议 5-8 页
(C) 10-20 分钟，建议 8-12 页
(D) 20 分钟以上，建议 12-18 页

### 第 3 轮：叙事与节奏

**Q8: 偏好什么叙事顺序？**
推荐选项根据 Q1 动态调整：
- (A) 结论先行（先抛结论，再展开论证）
- (B) 时间线叙事（按时间顺序讲述发展）
- (C) 问题-方案-验证（经典提案结构）
- (D) 对比分析（多方案并排比较）
- (E) 渐进呈现（从基础到复杂，逐步建构知识）
- (F) 故事弧线（现状-理想-跨越，情感驱动）

根据 Q1 标注推荐：B/C类目的→A，C类→E，D类→F。

**场景触发（仅当特定条件满足时追问）：**

T1（仅当 Q3=A 且 Q2=A 或 D）：会场屏幕比例？(A) 16:9 **←推荐** / (B) 4:3 / (C) 不确定。有无提词器？(A) 无 **←推荐** / (B) 有 / (C) 不确定。

T2（仅当 Q2=D）：需要多语言版本吗？(A) 仅中文 **←推荐** / (B) 中英双语 / (C) 纯英文。

### 汇总确认

全部问题完成后，输出如下汇总：

```
根据你的回答，推荐方案：

目的：[答案]
受众：[答案]
场景：[答案]
核心信息：[用户输入]
素材状态：[答案]
品牌约束：[答案]
页数：[答案或默认值]
叙事结构：[答案或推荐值]
配色方案：[business_blue 或其他]
预计输出格式：[PPTX/PDF等]

确认以上方案？回复"确认"开始生成规格，或指出需要调整的项。
```

## 规格生成规则

用户确认后，根据以下映射规则生成 Markdown 规格文档。

### 目的 → 幻灯片类型配比

| 目的 | 结构配比 |
|------|----------|
| B 说服 | cover + 结论页 + 3-4页论证 + 1-2页数据 + summary + contact |
| A 汇报 | cover + 4-5页内容 + 1页指标 + summary |
| C 教学 | cover + agenda + 4-6页知识模块 + summary |
| D 品牌 | cover + 2-3页核心信息 + 0-1页数据 + 1-2页图片展示 + summary |

### 受众 → 信息密度

| 受众 | 每页要点数 | 文字风格 |
|------|:--------:|----------|
| A 高管 | <=3条 | 标题即结论，每条<=15字 |
| B 中层 | 3-5条 | 标题结论+支撑点 |
| C 一线 | 5-8条 | 可密集，操作导向 |
| D 外部 | 3-4条 | 禁止内部术语，加背景解释 |

### 场景 → 格式参数

| 场景 | 输出格式 | 字号底线 | 配色倾向 |
|------|----------|:------:|----------|
| A 投影 | PPTX | 18pt | business_blue / tech_dark |
| B 附件 | PPTX + PDF | 14pt | business_blue / minimal_gray |
| C 打印 | PPTX + PDF | 14pt | academic_white / minimal_gray |
| D 移动端 | PPTX | 24pt | 任一，极简优先 |

### 叙事 → 页面顺序

| 叙事 | 页面编排 |
|------|----------|
| A 结论先行 | cover → 结论 → 论证1 → 论证2 → 论证3 → 数据支撑 → summary → contact |
| B 时间线 | cover → 起点 → 里程碑1 → 2 → 3 → 当前→未来 → summary |
| C 问题方案 | cover → 问题 → 方案概述 → 方案细节 → 验证/案例 → summary → contact |
| D 对比 | cover → 维度说明 → A方案 → B方案 → C方案 → 对比表 → 推荐 → summary |
| E 渐进呈现 | cover → agenda → 基础 → 进阶 → 高级 → 要点回顾 → summary |
| F 故事弧线 | cover → 现状 → 理想 → 障碍 → 跨越方案 → 新常态 → summary → contact |

### 跨维度冲突解决

当多个维度推荐冲突时，按以下优先级：
1. 场景优先于叙事（物理格式约束不可妥协）
2. 受众优先于目的（同一内容，给高管必须精简，给分析师可以密集）
3. 品牌约束优先于默认推荐
4. 页数上限优先于内容野心

## 输出规格格式

确认后输出以下 Markdown 规格文档（复制到任何 PPT 工具即可执行）：

```markdown
# PPT 规格文档

## 全局参数
- 配色方案: [style]
- 页数: [N]页
- 输出格式: [format]
- 屏幕比例: [ratio]
- 字号底线: [size]pt
- 叙事结构: [structure]

## 每页规格

### 第1页 | cover
- 标题: [核心信息]
- 副标题: [组织名/日期]
- 变体: centered

### 第2页 | [type]
- 标题: [每页一个观点]
- 内容要点: [根据信息密度约束列出]

...（逐页列出所有 slides）
```

## 生成后：Skill 链自动衔接

规格输出后，不要只丢一句"请找 PPT 工具执行"。自动扫描本地已安装的 PPT 相关 skill，帮助用户完成最后一步。

### Step 1：扫描本地 PPT Skill

扫描当前平台 skills 目录下所有与 PPT 生成相关的 skill。搜索关键词：目录名包含 `pptx`、`ppt`、`slide`、`deck`、`presentation`，且排除自身（`guided-pptx`）。

对每个找到的 skill，读取其 SKILL.md 的 `name` 和 `description` 字段，了解其输入格式和能力边界。

### Step 2：动态列出可用选项

将扫描结果展示给用户。不要预设任何 skill 名称——每次运行时动态生成列表：

```
本地找到以下可执行 PPT 生成的 skill：

(A) <skill-name-1> — <从 SKILL.md description 字段提取的一句话描述>
(B) <skill-name-2> — <从 SKILL.md description 字段提取的一句话描述>
(C) 不调用任何 skill —— 仅保留规格文档，我稍后自己处理
```

同时标注：哪些 skill 支持 JSON 输入（结构化程度高，规格可直接映射），哪些是自然语言驱动（规格需整合成描述性文字）。这个判断通过阅读每个 skill 的 SKILL.md 正文来判断，不硬编码。

### Step 3：格式适配

根据用户选择的 skill，动态判断其输入格式并转换：

**判断方法**：阅读目标 skill 的 SKILL.md，寻找其描述的输入方式——如果提到 JSON schema、结构化字段，则按 JSON 格式输出；如果提到"描述""主题""需求"等自然语言入口，则将规格整合为自然语言描述。

**通用转换策略**：

1. 如果目标 skill 接受 JSON 输入 → 将 Markdown 规格转为 JSON。使用目标 skill 自己定义的 schema（从它的 SKILL.md 中提取），将规格中的核心信息、页面大纲、配色方案、叙事结构映射到对应字段。

2. 如果目标 skill 接受自然语言输入 → 将规格整合为一段结构化描述，包含：核心信息、受众、场景、页数、叙事结构、每页的标题和要点。

3. 如果无法判断目标 skill 的输入格式 → 保持 Markdown 规格原文，同时附注：「目标 skill 的输入格式未识别，以下是原始规格，请根据需要调整后交给 <skill-name>」。

4. 用户选"不调用" → 保留 Markdown 规格原文，提示：「规格已保存在当前目录，随时可以交给任何 PPT 工具使用。」

### Step 4：执行与校验

1. 将格式化后的输入交给目标 skill 执行
2. 生成完成后检查：页数是否匹配？核心信息是否在 cover/summary 中体现？
3. 如有偏差，报告差异点而不是直接修改

### Skill 链设计原则

1. **只衔接不重复**——guided-pptx 不做 PPT 生成，只做引导+衔接。生成逻辑完全交给下游 skill
2. **不假设 skill 一定存在**——如果扫描不到任何 PPT skill，优雅降级为输出 Markdown 规格 + 手动建议
3. **不自动选择**——即使只有一个 PPT skill，也向用户确认后再调用。用户可能想只用规格不用工具
4. **保持规格可回退**——无论是否调用下游 skill，Markdown 规格始终保留在工作目录，用户可手动使用
