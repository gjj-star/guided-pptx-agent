# 最少必要问题集

从 19 个专家问题压缩到 8 个核心问题 + 2 个场景触发问题。每个问题的答案可推断或消解多个原始问题。

---

## 压缩原则

1. **直接推断**：A 的答案可以直接推断 B，B 不需要单独问
2. **默认推荐**：可以不问用户，给出推荐默认值（用户可覆盖）
3. **场景触发**：只在特定条件下才需要确认
4. **合并同类**：多个问题是同一决策的不同维度，合并为一个结构化问题

---

## 依赖关系图

```
Q1 目的 ──→ 推断 Q7 (CTA)
    └──→ 推荐 Q10 (叙事结构)
    └──→ 触发 Q17 (互动环节)

Q2 受众 ──→ 推断 Q8 (演示者风格)
    └──→ 约束 Q10 (叙事结构细节)
    └──→ 触发 Q15 (色盲/可访问性)

Q3 场景 ──→ 推断 Q12 (交付格式)
    └──→ 约束 Q9 (字号/信息密度)
    └──→ 触发 Q16/Q18/Q19 (大型演讲/打印/动画)

Q5 素材 ──→ 合并 Q11 (数据敏感度)
    └──→ 触发 Q14 (多语言)

Q6 品牌 ──→ 合并色彩禁忌
```

## 每个问题保留的理由

### C1 目的
不可推断。同一组数据和素材，说服 CEO 批准预算和培训新员工是完全不同的 PPT。目的锚定所有后续决策的方向。

### C2 受众
不可推断。受众的身份、规模、认知水平是变量而非常量。给 5 个 VP 看 vs 给 500 个员工看，设计方案差异巨大。

### C3 场景
不可推断。物理约束——投影需要 18pt+ 字号和高对比度，邮件附件需要自解释能力，手机需要极简单页。这些不能靠猜。

### C4 核心信息
不可推断且不可跳过。如果用户说不出，说明内容还没准备好——此时应该建议用户先梳理内容，而不是开始做 PPT。这是 McKinsey 的经典把关原则。

### C5 素材状态
不可推断。从零梳理和已有文稿的差距是 3-5 倍工作量。这也是用户预期的校准点——素材不成熟时，需要更多轮次的协作。

### C6 品牌约束
不可推断但容易被用户忽略。大型企业的品牌合规返工是最高频的灾难场景。合并颜色禁忌以在一个问题中覆盖。

### C7 页数/时长
可通过场景推荐默认值（15 分钟→6-8 页），但用户的硬性限制必须确认。推荐默认值只是在用户说"没限制"时给的 fallback。

### C8 叙事节奏
原因是"不同受众+不同目的对应不同最优叙事策略"的映射不是一对一的。用户可能在多个合理选项中有偏好。提供推荐默认值 + 可选项设计。

## 问题流设计概要

```
Start
  │
  ▼
Batch 1（4 问）: C1 目的 / C2 受众 / C3 场景 / C4 核心信息
  │
  ▼
Batch 2（2-3 问）: C5 素材状态 / C6 品牌约束 / C7 页数时长
  │
  ▼
Batch 3（1-2 问）: C8 叙事节奏
  │ 可选: +T1/T2 场景触发
  │
  ▼
汇总确认: "根据你的回答，我推荐：[默认方案]。是否确认？"
  │
  ▼
生成规格 → 调用 pptx-generator
```

总计：2-3 轮 AskUserQuestion，每轮 2-4 问。用户全程在选选项，不需要写任何 prompt。
